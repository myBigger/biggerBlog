## 你有没有遇到过这些情况

同一个 HTTP 接口，别人调用 1 秒返回结果，你的程序跑要 5 秒。代码逻辑一模一样，网络带宽也没问题，但就是慢。

换了网络环境，加了代理之后，所有请求全部超时。代码一行没改，但程序就是连不上。

你要导出 2000 条记录，想做个并发加速，结果并发一开，请求反而开始排队，后面的请求越等越久，最终超时失败。

这些问题在技术面试里很少被问到，但在实际项目里出现频率极高。它们不是网络问题，不是服务器问题，而是对 Qt 网络栈理解不够深入导致的性能陷阱。

上期「必哥手记」讲了 Qt5 和 Qt6 的 SSL 后端差异，这期我们把目光从底层往上挪一挪，聊一聊日常开发中 QNetworkAccessManager 为什么慢、为什么卡、为什么在某些网络环境下直接罢工。

---

## 一、QNetworkAccessManager 生命周期陷阱

先从一个我见过的最常见的写法开始。

很多开发者的 Qt 网络请求代码是这样的：写一个工具函数，每次需要发请求的时候调用它，函数内部 new 一个 QNetworkAccessManager，发完请求，函数结束，manager 随着函数栈一起析构。功能测试完全正常，单个请求响应及时，数据解析正确。

但是一旦上了生产环境开始高频调用，问题就来了。

QNetworkAccessManager 不是一个轻量对象。Qt 框架在构造它的时候，会初始化一套完整的网络栈，包括线程池、连接管理器、SSL 配置等。这套初始化成本不低，但真正的问题出在析构阶段——当 manager 被销毁时，它需要关闭所有挂起的连接，清理内部状态，如果信号槽连接不当，还会触发一些意想不到的清理逻辑。

更关键的是，每次新建 manager 都会重新建立 SSL 会话。SSL 握手是网络请求中耗时的大头之一，建立一个新连接的 RTT（Round-Trip Time）开销远大于复用已有连接。而如果你每次请求都 new 一个 manager，等于每次都在做完整的 SSL 握手，这个开销在高并发场景下是致命的。

正确的做法是把 QNetworkAccessManager 作为服务层的单例或者长生命周期成员变量来管理。让它一直活着，复用底层的连接池和 SSL 会话。第一个请求建立连接和握手，后续请求直接复用。这个改动不需要改任何业务逻辑，只需要把变量的作用域从局部函数升级到类级别。

```cpp
// ❌ 错误写法：每次请求都新建 manager
void fetchUserData(const QString& userId) {
    QNetworkAccessManager mgr;  // 每次构造/析构，SSL 握手重来
    QNetworkRequest request(QUrl("https://api.example.com/user/" + userId));
    mgr.get(request);
    // ...
}

// ✅ 正确写法：复用 manager
class NetworkService {
private:
    QNetworkAccessManager m_manager;  // 长生命周期，复用连接池
};
```

很多人觉得这种写法是"过度设计"，觉得"反正功能测试能跑就行"。但性能问题从来不会在开发环境暴露，它只会在凌晨三点用户打电话投诉的时候准时出现。

## 二、连接池与并发限制

理解了 QNetworkAccessManager 的生命周期问题之后，再来看并发限制。

HTTP/1.1 协议里有一条不太被注意的建议：单台服务器（同一个 host）同时建立的连接数不宜超过 2 个。这不是硬性规定，更像是互联网世界的默契——大家都遵守，秩序就稳定，谁要是开几百个并发连同一台服务器，就是在破坏游戏规则。

Qt 默认配置下，QNetworkAccessManager 对单个 host 的并发请求上限是 6 个。这个数字在普通场景下完全够用——你做一个 CRUD 应用，同时发三四个请求已经算多的了，6 个上限根本碰不到。

但问题出在批量数据拉取场景。

假设你要实现一个数据导出功能，用户点了导出按钮，后端没有批量导出 API，你只能循环调 2000 次获取单条记录的接口。按正常逻辑，你把这 2000 次请求并发发出去，让它们同时跑，多线程加速。

如果你真的这样做了，你会发现前 6 个请求立刻发出去了，然后从第 7 个请求开始排队——因为 Qt 限制单个 host 只能同时有 6 个连接在用。第 7 个请求要等前 6 个里有任何一个完成才能发出去，第 8 个、第 9 个同理。以此类推，整体耗时并没有因为并发而减少，反而因为排队机制变得不可预测。

更隐蔽的问题是，很多开发者以为给每个请求 new 一个 QNetworkAccessManager 就能突破并发限制。我见过有人这么写：循环里每次请求都新建一个 manager，逻辑是"每个请求有自己的 manager，这样就能绕过 6 个限制"。

但这是对 Qt 架构的误解。QNetworkAccessManager 是应用层抽象，底层的 socket 池是进程级别的，多个 manager 共享同一套连接管理逻辑。结果就是想绕开的限制实际上并没有绕过，反而因为创建了多个 manager 增加了初始化开销，赔了夫人又折兵。

解决这个问题需要从架构层面入手，而不是在战术上找窍门。

如果后端有批量 API，首选方案是推动后端提供一次返回多条记录的接口，把 2000 次请求减少到 20 次甚至 2 次。这是投入产出比最高的解法。

如果后端不支持批量 API，那就需要自己在客户端侧实现请求队列化：维护一个队列，控制在并发数量（比如同时跑 4 个请求），每次请求完成之后从队列取下一个发出去。这个方案不依赖后端改造，客户端侧就能落地。

我自己在 QtAuthNet 里实现过这套队列机制，核心思路其实不复杂：用一个信号量控制并发数量，finished 信号的槽函数里触发下一个请求。整个模式是一次性的开发投入，之后所有需要并发控制的场景都可以直接复用。

## 三、代理配置：为什么加了代理就挂了

这是 Qt 网络编程里最容易被忽视的坑，而且坑得很隐蔽——代码一行没改，换了个网络环境就不工作了。

Qt 本身支持系统代理自动发现，在大多数场景下开箱即用。但这个能力在不同平台上的表现差异很大。

在 Windows 上，Qt 会读取系统代理设置。大多数情况下这个机制是可靠的，企业内网用户配置了系统代理，Qt 自动就能识别。但问题出在某些企业内网使用了 PAC（Proxy Auto-Configuration）脚本的场景。Qt 对 PAC 的支持并不完整，有时候会静默失效——也就是说它不会报错，只是忽略了 PAC 配置，然后请求直接走直连。在客户的防火墙策略下，直连请求会被直接拒绝，于是你就看到"加了代理但请求还是挂了"的现象。

更复杂的问题是 SOCKS5 代理。

很多人以为配置了 SOCKS5 代理就能让所有流量走代理隧道，这其实是个误解。SOCKS5 代理在设计上支持两种模式：一种是让代理服务器解析 DNS 并建立连接（推荐做法），另一种是本地解析 DNS 只把 IP 地址发过去。Qt 的 SOCKS5 实现选择了后者。

这个差异在大多数场景下不会体现出来，但一旦进入企业内网环境，就会碰到一个经典问题：DNS 分割视图。什么意思呢？公司内部有很多内部域名，比如 `git.internal.corp.com`，这些域名只有公司 DNS 服务器才有解析记录，公网 DNS 完全不知道它们的存在。本地解析这些域名会直接失败，请求自然也就挂了。

HTTPS 请求走 HTTP 代理也有讲究。标准做法是建立 CONNECT 隧道，让代理服务器原样转发加密流量——代理只负责建立到目标服务器的 TCP 连接，之后的 HTTPS 握手发生在客户端和目标服务器之间，代理根本看不到加密内容。但 Qt 在某些版本的实现在 CONNECT 隧道的处理上存在兼容性问题，尤其是面对某些企业级代理软件（比如 Charles、Fiddler 的抓包模式）时表现不稳定。

这些问题没有银弹，每个场景的根因可能都不一样。但核心原则是清晰的：不要在开发环境完全直连、部署到客户环境才发现代理问题。在开发阶段就用一套跟目标环境一致的代理配置来测试，包括开发机也走代理，这是最低成本的防坑手段。

另一个建议是，如果应用面向企业内网用户，在界面上给用户留一个手动代理配置的入口。不是所有用户都有能力自己排查代理问题，但一个有代理配置界面的应用至少给了他们一个解决问题的抓手。

## 四、性能调优：从原理到实践

理解了 QNetworkAccessManager 的内部机制之后，性能调优的方向其实很清晰——不是对着配置项一顿乱改，而是知道在什么场景下该调什么参数。

**第一层是超时配置。**

Qt 默认的超时时间相当宽松。在网络条件好的环境下这不是问题，100 毫秒就能往返一次，超时设置成 30 秒还是 60 秒用户完全感知不到。但一旦进入弱网环境——移动网络跨国请求、信号不稳定的 Wi-Fi、客户机房里老旧的交换机——超时等待就会直接影响用户体验。

合理设置超时时间，配合有限次重试机制，能显著提升应用在复杂网络环境下的鲁棒性。但要注意的是，超时时间不是越短越好，太短了会在网络抖动时误判为失败，太长了会让用户等太久没有反馈。实践中我一般先设一个相对保守的初始值（比如 15 秒），然后在重试逻辑里逐步缩短——第一次等久一点给网络留足恢复时间，后续重试可以更快放弃。

**第二层是连接复用。**

HTTP 的 Keep-Alive 机制允许在同一个 TCP 连接上发送多个请求，避免每次请求都重新建立连接和握手。Qt 默认是启用这个特性的，但有一个前提条件：只有在同一个 QNetworkAccessManager 实例上发出的请求才能复用连接。

这就是为什么我一直强调 manager 要复用而不要每次新建。每次 new 一个 manager 出来，等于把 Keep-Alive 的能力完全放弃了——第一个请求建立连接，第二个请求又 new 了新的 manager，没有连接可复用，只能重新建立。看起来只是变量作用域的改动，实际效果是对性能的影响可能是 2 到 5 倍的差异。

**第三层是数据压缩。**

现代 HTTP API 大多支持 gzip 压缩传输，在响应头里声明 `Accept-Encoding: gzip`。开启解压支持能有效减少网络传输量，对移动网络或者跨国网络场景尤为明显。开启方式也不复杂，Qt 的网络请求默认会声明 gzip 支持，收到压缩响应后也会自动解压。但如果后端 API 没有开启压缩，这个就无能为力了——所以选 API 的时候这也是一个可以问后端的问题。

**第四层是 DNS 缓存。**

Qt 本身不提供显式的 DNS 缓存机制，但这不意味着 DNS 查询开销可以忽略。每次建立新连接都需要解析域名，对于高频请求同一个 API 的场景，DNS 查询的开销累积起来也是一笔账。

间接的优化思路是减少短时间内对同一域名的重复解析。通过合理的请求调度——比如把对同一域名的请求尽量集中在一起、避免随机打散——能有效降低 DNS 查询的频率。在 QNetworkAccessManager 的设计里，只要 manager 是复用的，底层的 socket 管理就会自动处理连接的复用和生命周期，其中就包括了对 DNS 解析结果的复用。

调优不是一个"找到正确答案然后照着做"的过程，而是一个"理解系统行为然后做出判断"的过程。每加一个调优参数，都要问自己：这个参数改完之后，在什么场景下会变好，在什么场景下会变差？tradeoff 是什么？只有在你能回答这些问题的时候，这个改动才是有意义的。

## 五、常见坑：那些让人抓狂的细节

除了前面提到的几类问题，还有几个高频踩坑点值得单独提醒。

**内存泄漏。** QNetworkAccessManager 的 finished 信号如果不正确处理，特别是使用 lambda 表达式捕获临时对象时，容易在高频请求场景下造成内存泄漏。manager 自己析构了，但信号槽连接还留着，生命周期管理一乱，内存就慢慢涨上去了。解决方法也不难：用 `qDeleteAll` 配合 `deleteLater` 处理响应数据，lambda 捕获尽量用值捕获而非引用，避免悬挂指针。

**跨线程使用。** 这是 Qt 里的经典问题，但每年都有人踩。QNetworkAccessManager 必须在创建它的线程里使用，所有信号槽也会在创建线程的 event loop 里分发。如果你在主线程创建了 manager，却在工作线程里调用它的方法，行为是未定义的——有时候能跑，有时候不报错但数据丢了，有时候直接崩溃。错误信息可能很模糊，让你排查半天也找不到根因。解决方法很简单：永远在工作线程里创建和使用 manager，或者统一在主线程处理。

**SSL 配置的继承逻辑。** QNetworkAccessManager 有一个默认的 SSL 配置，但这个配置不会自动继承到每个新创建的请求上。如果你在 manager 层面改了 SSL 配置，以为所有请求都会生效，结果可能让你失望——某些情况下请求仍然用的是默认配置。需要在请求级别显式设置 SSL 配置，或者确保 manager 的默认配置已经被正确更新。

这些问题单独拎出来看都是细节，但组合在一起就能把一个本来稳定的系统变得莫名其妙地出问题。Qt 网络编程的门槛不高，但上限不低——上手容易，要用得稳、用得高性能，确实需要对它的设计逻辑有更深入的理解。
